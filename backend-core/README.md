# Kajian App core platform backend

Owns everything the two ASR inference workers (`../backend/` for
Qwen3-ASR, `../backend-whisper/` for Whisper large-v3) deliberately
don't: users, kajian sessions/transcripts/notes, and audio storage. Those
two stay pure, stateless inference containers â€” this service is where
"who owns what" actually lives.

- **Auth**: the app already signs in via Firebase Auth (Google/Apple)
  client-side; this service verifies the Firebase ID token on every
  request and auto-provisions a local `User` row on first sight. Firebase
  stays the identity provider â€” this service owns the user's actual data.
- **Storage**: Postgres for structured data (sessions, transcript
  segments, notes, scripture references), MinIO (or any S3-compatible
  store) for audio files. The API never proxies raw audio bytes through
  itself â€” it mints short-lived presigned upload/download URLs and the
  app talks to the object store directly.
- **Processing**: `POST /sessions/{id}/transcribe` proxies to whichever
  ASR worker you ask for (`qwen` or `whisper`) and persists the result;
  `POST /sessions/{id}/summarize` calls Anthropic directly (server-side
  key) and persists the generated notes. This replaces the app's own
  `SessionProvider.process()` pipeline, which used to call
  `CloudTranscriptionService`/`AiNotesService` directly from the device.
- **Live captions**: `WS /transcribe/stream` relays the app's live-recording
  audio to the Qwen worker (the only one of the two with a streaming
  story) for low-latency captions during recording, the WebSocket
  counterpart to the REST proxy above. The app connects only to this
  endpoint â€” never to `../backend/`'s own `/transcribe/stream` directly â€”
  so the worker's URL/token stay a server-side secret. See
  `app/routers/streaming.py`.
- **Admin**: `routers/admin.py` backs a separate Next.js app (`../admin/`)
  â€” user list, per-user session browsing with audio playback, basic
  usage stats. See `scripts/promote_admin.py` for granting admin access
  (deliberately not exposed via the API itself).

## Setup (Docker, recommended)

```bash
cd backend-core
cp .env.example .env
# edit .env â€” at minimum set FIREBASE_SERVICE_ACCOUNT_PATH,
# ANTHROPIC_API_KEY, and the two ASR backend URLs

docker compose up --build -d
docker compose logs -f
```

You'll need a Firebase service account key mounted into the container at
whatever path `FIREBASE_SERVICE_ACCOUNT_PATH` points to (get one from the
Firebase console: Project Settings > Service Accounts > Generate new
private key) â€” add a `volumes:` bind mount for it in `docker-compose.yml`,
e.g. `./firebase-service-account.json:/run/secrets/firebase-service-account.json:ro`.

Health check:

```bash
curl http://localhost:8090/health
# {"status":"ok"}
```

Grant yourself admin access after signing into the app at least once:

```bash
docker compose exec kajian-core-api python scripts/promote_admin.py you@example.com
```

## Setup (plain Python, no Docker)

Requires a running Postgres and MinIO (or point `S3_ENDPOINT_URL` at a
real S3/R2 bucket instead).

```bash
cd backend-core
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt

cp .env.example .env  # edit as needed
export $(grep -v '^#' .env | xargs)
alembic upgrade head
uvicorn app.main:app --host 0.0.0.0 --port 8080
```

## Running tests

These are integration tests against a **real Postgres** (not a mocked
DB) â€” the bugs worth guarding against here (SQLAlchemy identity-map
staleness after certain ORM mutation patterns) only reproduce against a
real session/DB round-trip. Firebase and the object store are stubbed
out (dev auth bypass + an in-memory fake), so Postgres is the only
external dependency.

```bash
cd backend-core
docker run -d --name kajian-core-test-pg \
  -e POSTGRES_USER=kajian -e POSTGRES_PASSWORD=kajian -e POSTGRES_DB=kajian \
  -p 55432:5432 postgres:16-alpine

python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt pytest anyio httpx

export DATABASE_URL="postgresql+asyncpg://kajian:kajian@localhost:55432/kajian"
export CORE_DEV_AUTH_BYPASS=true
alembic upgrade head
pytest -v

docker rm -f kajian-core-test-pg  # when done
```

## Database migrations

Uses Alembic. The initial schema is a hand-written migration (not
autogenerated) at `app/migrations/versions/0001_initial_schema.py` â€” when
adding new migrations, `alembic revision --autogenerate -m "..."` works
normally against a running DB with the previous migration already
applied.

## Pointing the Flutter app at this backend

```bash
flutter run --dart-define=BACKEND_BASE_URL=http://<your-homelab-host>:8090
```

This becomes the app's single backend URL going forward â€” it internally
proxies to the Qwen/Whisper workers rather than the app talking to them
directly (see `docs/BACKEND.md` in the Flutter repo for the full picture
of how this fits with the two ASR backends).

## API surface

| Route | Auth | Purpose |
|---|---|---|
| `GET /me` | user | Current user's profile |
| `GET/POST /sessions` | user | List / create own sessions |
| `PATCH/DELETE /sessions/{id}` | user | Update / delete own session |
| `PUT /sessions/{id}/transcript` | user | Replace a session's transcript wholesale |
| `PUT /sessions/{id}/note` | user | Replace a session's AI notes wholesale |
| `POST /sessions/{id}/audio-upload-url` | user | Get a presigned upload URL |
| `POST /sessions/{id}/audio-confirm` | user | Mark audio as uploaded |
| `GET /sessions/{id}/audio-url` | user | Get a presigned download URL |
| `POST /sessions/{id}/transcribe` | user | Run the ASR proxy against uploaded audio |
| `POST /sessions/{id}/summarize` | user | Generate AI notes from the transcript |
| `WS /transcribe/stream` | user (`?token=`) | Live captions during recording, relayed to the Qwen worker |
| `GET /admin/stats` | admin | Basic usage stats |
| `GET /admin/users` | admin | All users + session counts |
| `GET /admin/users/{id}/sessions` | admin | Any user's sessions |
| `GET /admin/sessions/{id}/audio-url` | admin | Any session's audio (support/moderation) |
